# Migration v2.1.5 - PRICEMATRIX Index Optimization - Developer Runbook

## Overview
This runbook provides step-by-step instructions for executing the v2.1.5 database migration that adds composite indexes to the PRICEMATRIX table.

**Performance Impact**: Reduces discount calculation queries from 552ms to 2-5ms (100x faster)

---

## Migration Files

### 1. `migration.sql` - Production/Ubuntu/Linux (UPPERCASE table names)
- **Use for**: Production, QA, Ubuntu/Linux environments
- **Table name**: `PRICEMATRIX` (uppercase)
- **Execution**: Run through migration system or mysql command line

### 2. `migration-dev.sql` - Development/Windows (lowercase table names)
- **Use for**: Development, Windows environments
- **Table name**: `pricematrix` (lowercase)
- **Execution**: Run manually through mysql command line or modify migration system config

---

## Why Two Versions?

MySQL table names are **case-sensitive on Linux/Unix** but **case-insensitive on Windows**:
- **Production (Ubuntu/Linux)**: Uses `PRICEMATRIX` (uppercase)
- **Development (Windows)**: Uses `pricematrix` (lowercase)

The JPA/JDBC migration system executes SQL statements independently, making it difficult to use dynamic SQL with session variables. Therefore, we provide two separate migration files.

## Verification

After running migration, verify indexes were created:

```sql
-- Check indexes exist
SELECT TABLE_NAME, INDEX_NAME, COLUMN_NAME, SEQ_IN_INDEX, CARDINALITY
FROM INFORMATION_SCHEMA.STATISTICS
WHERE INDEX_NAME LIKE 'idx_psd_%'
ORDER BY TABLE_NAME, INDEX_NAME, SEQ_IN_INDEX;

-- Expected result: 3 indexes (idx_psd_item, idx_psd_category, idx_psd_department)
-- Each with 5 columns
```

Test query performance:

```sql
-- Should use idx_psd_item index
EXPLAIN SELECT DISCOUNTPERCENT
FROM pricematrix  -- or PRICEMATRIX on production
WHERE RETIRED=0
  AND PAYMENTMETHOD='Cash'
  AND PAYMENTSCHEME_ID IS NULL
  AND MEMBERSHIPSCHEME_ID IS NULL
  AND ITEM_ID=1;

-- Look for "Using index" in Extra column
-- rows examined should be low (< 100)
```

**Expected Query Plan**:
- **type**: `ref` or `const`
- **possible_keys**: `idx_psd_item`
- **key**: `idx_psd_item`
- **Extra**: Should include `Using index`
- **rows**: < 100 (typically < 10)

## Rollback

If needed, drop the indexes:

```sql
-- For Production (uppercase)
DROP INDEX IF EXISTS idx_psd_item ON PRICEMATRIX;
DROP INDEX IF EXISTS idx_psd_category ON PRICEMATRIX;
DROP INDEX IF EXISTS idx_psd_department ON PRICEMATRIX;

-- For Development (lowercase)
DROP INDEX IF EXISTS idx_psd_item ON pricematrix;
DROP INDEX IF EXISTS idx_psd_category ON pricematrix;
DROP INDEX IF EXISTS idx_psd_department ON pricematrix;
```

**Note**: Application will continue to work without indexes (just slower). Rollback is safe and non-destructive.

## Troubleshooting

### Issue: "Table 'PRICEMATRIX' doesn't exist"
**Solution**: You're likely on Windows/Development - use `migration-dev.sql` instead

### Issue: "Duplicate key name 'idx_psd_item'"
**Solution**: Indexes already exist - verify with:
```sql
SHOW INDEX FROM PRICEMATRIX WHERE Key_name LIKE 'idx_psd_%';
```

### Issue: Slow performance after migration
**Solution**: Verify indexes are being used with EXPLAIN query above

## Technical Details

### Indexes Created

1. **idx_psd_item** - Item-specific discount lookup
   - Columns: `RETIRED, PAYMENTMETHOD, PAYMENTSCHEME_ID, MEMBERSHIPSCHEME_ID, ITEM_ID`
   - Use case: Most specific discount (highest priority)

2. **idx_psd_category** - Category discount lookup
   - Columns: `RETIRED, PAYMENTMETHOD, PAYMENTSCHEME_ID, MEMBERSHIPSCHEME_ID, CATEGORY_ID`
   - Use case: Category-level discount (medium priority)

3. **idx_psd_department** - Department discount lookup
   - Columns: `RETIRED, PAYMENTMETHOD, PAYMENTSCHEME_ID, MEMBERSHIPSCHEME_ID, DEPARTMENT_ID`
   - Use case: Department-level discount (lowest priority)

### Query Optimization Strategy

The discount lookup follows a hierarchy:
1. Item-specific discount (most specific) → uses `idx_psd_item`
2. Category discount → uses `idx_psd_category`
3. Parent category discount → uses `idx_psd_category`
4. Department discount (least specific) → uses `idx_psd_department`

Each index is designed to match the exact WHERE clause pattern used in the application queries.
